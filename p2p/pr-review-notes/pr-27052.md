## `getpeerinfo`: add `last_block_announcement` field



## Understanding the Field

verify  understanding of what `last_block_announcement` tracks:

- Updates when a peer sends us a header we have **never seen** AND it has **more chain work than our tip**
- Does **NOT** update when a peer sends us a header we already have
- Does **NOT** update when a peer sends us a header with less work than our tip
- Starts at `0` for every new peer connection
- Used by stale tip eviction to decide which outbound full-relay peer to disconnect

**Distinction from `last_block`:**
- `last_block` — we received and saved a full valid novel block to disk
- `last_block_announcement` — peer told us a block exists via header or compact block

## 2. RPC Description Accuracy

Current description:
```
"The " + UNIX_EPOCH_TIME + " this peer was first to announce a block"
```

**Issues:**
- [ ] Does not mention the chain work condition — field only updates when header has **more work than our tip**
- [ ] "first to announce" implies a race between peers which is not exactly what is tracked — it is the last time **this peer** told us about a header we had not seen

**Suggested improvement:**
```
"The last time this peer announced a new block header with more work than our tip, or 0 if they never have"
```

## 4. Test Coverage Gaps

The test covers:
- [x] Field starts at `0` for a new peer
- [x] Field updates when peer sends a new block
- [x] Field does not update when peer sends a header for a block we already have

The test does **not** cover:
- [ ] Field does not update when peer sends a **genuinely new** header with **less work** than our tip
- [ ] Field updates correctly via the **compact block** path — only the headers path is tested
- [ ] Stale tip eviction correctly uses this field to choose which peer to disconnect
- [ ] Missing assertion — after sending a known header, `last_block` should also not update:

```python
self.log.info("Verify that block announcement time isn't updated")
peerinfo = node.getpeerinfo()[0]
assert peerinfo['last_block_announcement'] == cur_time
assert peerinfo['last_block'] == cur_time  # missing — should be added
```

### The Core Bug This Test Catches

Before PR #26172 the null check was flipped:
```cpp
// Before fix — wrong
bool received_new_header{last_received_header != nullptr};

// After fix — correct
bool received_new_header{last_received_header == nullptr};
```

This assertion would **fail before the fix**:
```python
# New block -> not in index -> LookupBlockIndex returns nullptr
# Flipped condition: nullptr != nullptr -> false
# -> received_new_header = false
# -> m_last_block_announcement never updates -> still 0
assert peerinfo['last_block_announcement'] == cur_time  # would fail, field is 0
```
